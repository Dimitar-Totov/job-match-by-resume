# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm run dev` — start the Vite dev server with HMR
- `npm run build` — type-check via `tsc -b` then build for production with Vite
- `npm run lint` — run Oxlint
- `npm run preview` — preview the production build locally

There is no test runner configured.

Running the app requires Supabase credentials: copy `.env.example` to `.env` and set `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` (`src/services/supabaseClient.ts` throws at import time if either is missing).

Database schema changes go through Supabase CLI migrations in `supabase/migrations/` (plain timestamped SQL files, no `supabase/config.toml` — the CLI was used unlinked/ad hoc via `supabase link --project-ref <ref>` then `supabase db push`). There's no local Postgres/CLI project scaffold beyond the migrations folder; write a new migration file rather than editing an applied one, and never run `supabase db push` without the user's explicit go-ahead since it applies directly to the shared remote project.

Server-side code (e.g. anything needing a third-party API key like `OPENROUTER_API_KEY`, which must never ship in the client bundle) goes in `supabase/functions/` as Supabase Edge Functions (Deno, `Deno.serve`, npm packages via `npm:pkg@version` specifiers) — the only server-side compute this project has. `supabase/functions/_shared/` holds the code shared across functions: `cors.ts` (CORS headers for browser-invoked functions), `guards.ts` (`withAiGuards()` — the CORS-preflight + in-function-auth + per-user-rate-limit preamble every AI function runs before spending tokens, described below), and `ai.ts` (`createAiClient()`/`getModel()` for the env-driven OpenRouter client, plus `str`/`arr`/`obj`/`stripFence` JSON-coercion helpers the parsers share). `supabase/functions/deno.json` is a minimal Deno config so the CLI/editor resolve npm specifiers and Deno globals (this directory is intentionally outside `tsconfig.app.json`'s `include` and is listed in `.oxlintignore`, so it doesn't affect `tsc -b`/`npm run lint`). Deploy with `supabase functions deploy <name>` and set secrets with `supabase secrets set OPENROUTER_API_KEY=...` — like `supabase db push`, never run either without the user's explicit go-ahead since they act on the shared remote project.

The AI provider is **OpenRouter** (not OpenAI directly): functions talk to it through the OpenAI npm SDK pointed at OpenRouter's OpenAI-compatible endpoint, defaulting to a **free** model so calls cost $0. Provider/model are fully env-driven so they stay swappable via secrets without a code edit: `OPENROUTER_API_KEY` (required), `AI_BASE_URL` (default `https://openrouter.ai/api/v1`), `AI_MODEL` (default a free slug — free slugs get retired periodically, so prefer overriding via the secret), and optional `AI_REFERER`/`AI_TITLE` OpenRouter attribution headers.

`supabase/functions/ai-health/` is the **template** to copy for any AI/third-party-key function — a cheap connectivity check (a tiny real chat completion, called from the client via `src/services/aiService.ts`'s `checkAiHealth()`). The token-billed resume functions follow it: `resume-parse/` downloads the uploaded PDF from Storage, extracts text with `unpdf`, asks the model for JSON matching `ParsedResume`, and upserts into `public.resumes`; `resume-analyze/` reads that stored `parsed` resume and asks the model for the `ResumeAnalysis` shape (see the resume section below). Each wraps its handler in `withAiGuards(req, { action, max, window }, handler)` (from `_shared/guards.ts`), which runs the mandatory preamble — in order, before spending tokens: (1) **authenticate the caller in-function** via `supabase.auth.getUser()` (`verify_jwt=true` alone is not enough because the public anon key is itself a valid signed JWT); and (2) **rate-limit per user** via the Postgres `check_rate_limit()` RPC (see "Rate limiting" below), since signup is self-serve and auth alone won't stop a registered user from hammering a token-billed endpoint — then hands the handler an authenticated `{ supabase, userId, jsonHeaders, req }` context. So a new AI function is just: `withAiGuards` with its own rate-limit budget + `createAiClient()` + its prompt/coercion/persistence. Handlers return the *result* (`{ ok, ... }`) as HTTP 200 so the client reads it from `data`, while genuine errors (401 unauth, 429 rate-limited, surfaced by the guards) stay non-2xx — `supabase.functions.invoke` maps any non-2xx to the `error` channel with `data: null`, so a 200-with-`ok:false` body is how a "handled failure" (e.g. an image-only PDF with no extractable text) reaches the client.

## Testing

Do not run tests (npm test, npx playwright test, vitest, jest, etc.) automatically after implementing a feature or fixing a bug. Only run tests when explicitly asked to.

## Architecture

Vite + React 19 + TypeScript SPA for an AI resume/job-match analyzer. There's no backend of our own — Supabase is used directly as the auth provider and, for the `profiles` table, as the database (see below). `react-router-dom` owns every URL in the app, including every in-shell screen as its own flat, top-level route (e.g. `/upload`, `/jobs/match`); a `Screen`-union façade (`useNav()`) sits on top of the in-shell routes to keep call sites simple (see below). No styling library or fetch library is used — don't introduce Tailwind or a fetch library without explicit instruction.

### Navigation is entirely router-driven, with a `Screen`-based façade inside the dashboard

`react-router-dom` (`BrowserRouter` in `src/main.tsx`) owns every route, defined in `src/App.tsx` via `PATHS` (`src/routes/paths.ts`) and `SCREEN_PATHS` (`src/features/app-shell/navConfig.ts`): `/` (welcome), `/register`, `/login`, `/onboarding`, `/dashboard`, and one flat top-level route per in-shell screen (e.g. `/upload`, `/review`, `/jobs/match`, `/generate/tailor`). Each pre-dashboard screen has a dedicated route-element component in `src/pages/` (`WelcomeRoute`, `RegisterRoute`, `LoginRoute`, `OnboardingRoute`) that owns the sign-up/sign-in submit handlers and their loading/error state — mirrors what the now-deleted `RootView.tsx` used to do in one file, just split per route. Email (welcome → register) and pending registration data (register → onboarding) travel via React Router's `navigate(path, { state })`, typed as `RegisterLocationState`/`OnboardingLocationState` with runtime type guards (no `any`) since `location.state` is `unknown`.

Every one of those routes is guarded, based on `useAuth().status` (see below):
- `RequireGuest` (`src/routes/RequireGuest.tsx`) wraps `/`, `/register`, `/login`, `/onboarding` — redirects to `/dashboard` if already authenticated.
- `RequireAuth` (`src/routes/RequireAuth.tsx`) wraps `DashboardRoute` (a pathless layout route in `App.tsx`, ancestor of `/dashboard` and every in-shell screen route) — redirects to `/login` if a guest.
- Both render `null` while `status === 'loading'` (the initial Supabase session check hasn't resolved yet) rather than redirecting, so neither an about-to-be-authenticated user nor a guest sees the wrong screen flash before the real status is known.
- An unmatched path falls through to a catch-all `<Route path="*">` that redirects to `/`.

Every sidebar screen has its own real, flat, deep-linkable, top-level URL (e.g. `/upload`, `/jobs/match`), defined directly in `src/App.tsx` as child `<Route>`s of a pathless `<Route element={<AppShell/>}>` layout route (itself nested inside the pathless, `RequireAuth`-guarded `DashboardRoute` layout route), and rendered inside `AppShell`'s `Sidebar`+`AppHeader` layout via `<Outlet/>`. `DashboardRoute` itself no longer owns any `<Route>`s — it's just `NavProvider` wrapping an `<Outlet/>`. Feature code never touches `react-router-dom` directly, though — it keeps using the pre-existing `Screen`-union façade: `NavContext`/`NavProvider` (`src/context/`) expose the current `Screen` (`src/types/index.ts`) and a `navigate(screen, options?)` function, plus sidebar-collapsed state, via the `useNav()` hook (`src/hooks/useNav.ts`) — never import `NavContext` directly in components (that split exists to satisfy Oxlint's `react/only-export-components`). Under the hood, `NavProvider` derives `screen` from the current URL (via `useLocation`, matched by exact pathname against `SCREEN_PATHS`, restricted to the in-shell subset of the `Screen` union) and `navigate()` looks up the target `Screen`'s path in `SCREEN_PATHS` and calls react-router's `navigate(path, { replace })` (via `useNavigate`) — so `navigate(screen, { replace: true })` still swaps the current history entry instead of pushing, and back/forward is handled entirely by the router now (no manual `pushState`/`popstate` code left). `navigate('welcome', ...)`, e.g. from Sidebar's logout, correctly leaves the authenticated subtree since `SCREEN_PATHS` covers the full `Screen` union, not just in-shell screens.

`src/features/app-shell/navConfig.ts` is the single source of truth for sidebar nav items (`NAV_SECTIONS`, `BOTTOM_NAV`), the per-screen page title (`SCREEN_TITLES`), and the per-screen URL (`SCREEN_PATHS`). Adding a new **in-shell** screen means: add it to the `Screen` union, add a section/page component under `src/features/<name>/`, add a `<Route>` for it in `App.tsx` (nested under the `AppShell` layout route), and add entries to `navConfig.ts` (`SCREEN_PATHS`, `SCREEN_TITLES`, and `NAV_SECTIONS`/`BOTTOM_NAV` if it should be reachable from the sidebar). Adding a new **top-level, pre-dashboard** route (rare) means: add a path to `PATHS`, a route-element component in `src/pages/`, a `<Route>` entry in `App.tsx` wrapped in the appropriate guard.

### Auth state

`AuthContext`/`AuthProvider` (`src/context/authContext.ts`, `src/context/AuthProvider.tsx`) track whether the current visitor is a guest or an authenticated user. `AuthProvider` wraps the whole app (outside the router's routes, inside `BrowserRouter`) and subscribes to `supabase.auth.onAuthStateChange`, treating its `INITIAL_SESSION` event as the source of truth (with a `getCurrentSession()` call as a seed for the brief window before that first event fires). Consume via `useAuth()` (`src/hooks/useAuth.ts`), which returns `{ status: 'loading' | 'authenticated' | 'guest', user, session }`. `status === 'loading'` is the window before the initial session check resolves — route guards (`RequireGuest`/`RequireAuth` above) must not redirect during that state. The Supabase client (`src/services/supabaseClient.ts`) is configured with `auth: { storage: window.sessionStorage }`, so sessions clear when the tab/window closes rather than persisting indefinitely like the SDK's `localStorage` default.

### Profile data (Supabase table, not a mock)

Unlike everything else pre-real-backend, per-user profile data is real: `supabase/migrations/20260718000000_create_profiles_table.sql` creates a `profiles` table 1:1 with `auth.users` (`id uuid primary key references auth.users(id) on delete cascade`), columns `username`, `email`, `field_of_study`, `graduation_year`, `target_roles text[]`, `target_industries text[]`, `experience_level`, `created_at`, `updated_at`. Row-level security is enabled with select/insert/update policies scoped to `auth.uid() = id` (no delete policy — cascades from `auth.users` instead); a `handle_new_user()` trigger on `auth.users` auto-inserts a starter row (`id`, `email`) on signup, and a trigger auto-touches `updated_at` on every update — callers never set it themselves. `src/services/profileService.ts` exposes `getProfile(userId)` / `updateProfile(userId, updates)`, following the same throw-the-raw-Postgrest/Auth-error convention as `authService.ts`; the `Profile` type lives in `src/types/index.ts` alongside the other domain types. `src/hooks/useProfile.ts` wraps `getProfile` in the same explicit-status-enum pattern as `useResumeParsing` (`idle | loading | done | error`, stale-response guarding via a request-id ref) and is the only current consumer, from `SettingsScreen.tsx` (see below) — follow it for any other screen that needs to read/write a user's own profile row.

### Resume data: parsing & analysis (Storage + Supabase table + Edge Functions)

Also real (no longer stubbed): `supabase/migrations/20260722000000_create_resumes.sql` adds a **private** `resumes` Storage bucket (RLS on `storage.objects` scoped so a user can only touch files under their own `<uid>/...` folder) and a `public.resumes` table 1:1 with `auth.users` (`user_id` primary key; columns `storage_path`, `file_name`, `parsed jsonb`, `status` (`pending | parsed | failed`), `created_at`, `updated_at`), owner-scoped RLS (select/insert/update, no delete), reusing the `set_updated_at()` trigger from the profiles migration. `parsed` holds the `ParsedResume` shape as jsonb (including an optional `summary` field). A follow-up migration adds AI-output columns to that same 1:1 row: `20260723000000_add_resume_analysis.sql` (`analysis jsonb`, `analysis_status` — `null | ready | failed`); the existing owner-RLS and `set_updated_at()` trigger already cover it, so a jsonb-column migration is all it takes. (A later migration, `20260724000000_add_resume_suggestions.sql`, added `suggestions`/`suggestions_status` columns for an AI rewrite-suggestions feature that has since been removed from the app; `20260725000000_drop_resume_suggestions.sql` drops them — never edit an applied migration, add a new one instead.) The `ResumeRecord`/`ResumeStatus`/`ResumeAnalysis` types live in `src/types/index.ts`.

The parse flow: `src/services/resumeService.ts` `parseResume(file, userId, onProgress, signal)` uploads the PDF to Storage under `<uid>/<uuid>.pdf`, upserts a `pending` row, then invokes the `resume-parse` Edge Function (which does the download/extract/AI-structure/persist server-side) and resolves with the `ParsedResume`. It throws the raw Supabase error on storage/invoke failure, or an `Error` (`'no_text'` for image-only PDFs / `'parse_failed'`) when the function reports it couldn't parse. The three-stage progress bar is a UI easing on a timer, not real backend progress. `getResume(userId)` reads the user's stored row (or null). `src/hooks/useResumeParsing.ts` drives `parseResume` (pulls `userId` from `useAuth()` and takes a `File`); `src/hooks/useResume.ts` reads the stored row following the `useProfile` status-enum pattern.

The Review screen (`src/features/review/ReviewScreen.tsx`) is a **read-only** display of that stored `parsed` resume, driven directly by `useResume`: this app matches an existing resume against jobs, it doesn't double as a resume builder/editor, so every section (contact, summary, education, skills, experience) renders as plain static text/chips — no inputs, no add/remove controls, no auto-save. "Confirm & analyze" just navigates to Analysis.

The analysis flow (in `resumeService.ts`, **lazy + cached on the row**): `analyzeResume()` invokes `resume-analyze` and resolves with a `ResumeAnalysis` (overall score, ATS checks, section feedback, writing issues, quick wins). It reads the already-stored `parsed` resume **server-side** — no re-upload — and persists its output onto the row, throwing `'no_resume'`/`'analyze_failed'` `Error`s on handled failure (mirroring `parseResume`). `useResumeAnalysis(userId)` (following the `useResume` status-enum + request-id pattern, extended with an `analyzing` and `empty` state) drives it: on mount it reads the row and either surfaces the cached analysis or auto-generates it once, so the *first* visit to the Analysis screen spends tokens but revisits don't. The screen gates on loading / empty (no parsed resume → upload CTA) / error states, and offers a manual re-analyze.

### Rate limiting (Postgres-backed, for token-billed Edge Functions)

`supabase/migrations/20260720000000_create_rate_limits_table.sql` adds a reusable per-user fixed-window limiter: a `rate_limits` table (RLS on with **no** policies — fully locked down, never user-readable/writable) plus a `check_rate_limit(p_action, p_max, p_window)` `SECURITY DEFINER` function that atomically records one call for `auth.uid()` in the current window and returns whether it's within budget (the `INSERT … ON CONFLICT DO UPDATE … RETURNING` is race-free). Edge Functions call it via `supabase.rpc('check_rate_limit', …)` after authenticating and deny on `false`. Different functions pass different `p_action` keys so their budgets are independent (`ai-health` 10/min; `resume-parse`, `resume-analyze` 10/hour each). Elapsed-window rows accumulate harmlessly (purge as a future scheduled job).

### Feature/component split

- `src/features/<name>/` — one folder per screen (welcome, register, login, onboarding, dashboard, upload, review, analysis, jobs, generate, skills, notifications, settings, app-shell). Each screen owns its own `.tsx` + co-located `.css`; some folders hold multiple related screens (e.g. `jobs/` has AddJob, Match, and Tracker; `generate/` has Tailor and CoverLetter).
- `src/components/` — shared presentational primitives (Button, Card, Badge, Chip, PillTag, Toggle, TextField, ProgressBar, ScoreRing, Icon, Logo), re-exported from `src/components/index.ts`. `Icon` wraps the Material Symbols Rounded font (loaded via Google Fonts `<link>` in `index.html`) — use it instead of adding an icon library. `TextField` auto-detects `type="password"` and renders a show/hide toggle internally — pass `type="password"` rather than building that toggle again in a screen.
- `src/services/` — `authService.ts` wraps Supabase Auth (`signUp`/`signIn`/`signOut`/`getCurrentSession`/`getCurrentUser`), throwing the raw Supabase `AuthError` on failure for callers to catch; `profileService.ts` wraps the `profiles` table (`getProfile`/`updateProfile`, see "Profile data" above), throwing the raw Postgrest error the same way; `resumeService.ts` (`parseResume`/`getResume`/`analyzeResume`, backed by Storage + the `resume-parse`/`resume-analyze` Edge Functions + the `resumes` table — see "Resume data" above), throwing the raw Supabase error the same way; `aiService.ts` (`checkAiHealth`, wraps the `ai-health` Edge Function, throwing the raw `FunctionsError`); `supabaseClient.ts` creates the singleton client from the `VITE_SUPABASE_*` env vars (configured to persist the session in `sessionStorage`, not the SDK's `localStorage` default). Auth, profile, and resume data (parsing and AI analysis) are now real; the remaining screens are still mock-backed: `mockData.ts` holds typed fixtures (jobs, the dashboard's score/checklist data, reference `parsedResume`/`analysisSections` shapes still read by the Dashboard, onboarding option lists like `onboardingFields`/`onboardingRoles`/`onboardingIndustries`/`onboardingExpLevels`/`onboardingGradYears`, etc.) consumed directly by screens until real data-fetching exists — `onboarding/` and `settings/` both import the same onboarding option lists from here so their chip choices stay in sync.
- `src/routes/` — router guards (`RequireGuest`, `RequireAuth`) and the `PATHS` constant; `src/pages/` holds the route-element components (`WelcomeRoute`, `RegisterRoute`, `LoginRoute`, `OnboardingRoute`, `DashboardRoute`). See "Navigation" above.
- `src/utils/` — small pure helpers: `cn.ts` (classname join, used instead of a `clsx`/`classnames` dep), `email.ts` (email validation), `score.ts` (score→level/color helpers).
- `src/hooks/useResumeParsing.ts` is the pattern to follow for any future async flow: explicit `idle | parsing | done | error` status, `AbortController` cleanup on unmount/restart, and a ref-captured `onComplete` callback to avoid stale closures. `src/hooks/useProfile.ts` and `src/hooks/useResume.ts` follow the same shape for fetching a Supabase-backed row (`idle | loading | done | error`, request-id staleness guard). `src/hooks/useResumeAnalysis.ts` extends the fetch pattern further — reading the row, then lazily generating + caching AI output when it's absent — see "Resume data" above.
- `src/types/index.ts` is the single domain-model file (Screen, Job, ParsedResume, ResumeRecord, ResumeAnalysis, MatchDimension, SkillGap, AppNotification, Profile, etc.) — add new domain types here rather than inlining them in feature files.
- `src/styles/tokens.css` holds the design system as CSS custom properties (colors, ink/panel/border scales, accent/green/amber/red status colors, shadows, radius) ported from the original design reference; `index.css`/`patterns.css`/`App.css` build on top of it. Never hardcode a color/shadow/radius a token already covers.

### TypeScript / lint conventions

- Solution-style TS config: `tsconfig.json` references `tsconfig.app.json` (app source, strict, `verbatimModuleSyntax: true` — always use `import type` for type-only imports) and `tsconfig.node.json` (Vite config). `src/vite-env.d.ts` declares the `ImportMetaEnv` shape for the `VITE_SUPABASE_*` vars — extend it there when adding new `VITE_`-prefixed env vars.
- Linting is via Oxlint (`.oxlintrc.json`), not ESLint — `react/rules-of-hooks` is an error, `react/only-export-components` is a warning (this is why context value objects and hooks live in separate files from provider components). Type-aware lint rules are not enabled.

## Subagents

`.claude/agents/frontend.md` and `.claude/agents/backend.md` define scoped subagents for this project — frontend owns the browser-facing layer only, backend owns server-side concerns only. There's still no custom backend server in this repo (Supabase is used directly from the client as a hosted auth provider and database, not through a backend we control); backend-agent work here means Supabase migrations/RLS policies and the `src/services/*Service.ts` data-access functions that wrap them (see `profileService.ts` above), not a server of its own. Each agent stays within its boundary and defers cross-boundary changes back to the user rather than improvising across the line.
